# SQLite API-call log

**Status:** Draft
**Date:** 2026-05-18

## Goal

Persist every successful QAP API call made by the `demo_App` server — both the request payload and the response body — into a local SQLite database (via `bun:sqlite`), so that data accumulated across multiple calls during a single session — and across all advisor/investor interactions — survives process restarts and can be inspected after the fact.

## Background

`select_and_lookup.ts` currently calls the QAP API through `DeepAlphaClient` (`public_api/client.ts`) from several handlers:

- `handlePick` — `listAdvisors`, optionally `createInvestor`
- `handleCreateSession` — `listInvestors`, optionally `createInvestor`, `createStateSession`
- `handleSessionCompleted` → `fetchSessionBundle` — `getAdviceSession`, `getAdviceInformation`, `listAdviceGoals`, `getAdviceGoal` (per goal), `getAdviceTransactions`, then `getReport` and `downloadReportPdf`

Today, only the post-completion bundle is persisted — as a merged `session_<id>_data.json` file plus a `session_<id>.pdf`. All other API responses are logged to stdout and lost on restart.

## Non-goals

- Migrations framework — schema is created with `CREATE TABLE IF NOT EXISTS` at boot.
- Logging failed API calls (4xx / 5xx / thrown exceptions). Only successful responses are recorded.
- Replacing the on-disk PDF artifacts. PDFs remain on disk; only their path/size is recorded.
- Read-side query API or dashboard. The DB is inspected ad-hoc via `sqlite3` / SQL clients.

## Design

### Schema

Two tables, one row per successful API response, in a single SQLite file at `data/qap.sqlite`.

```sql
CREATE TABLE IF NOT EXISTS session_api_calls (
  id            INTEGER PRIMARY KEY AUTOINCREMENT,
  session_id    TEXT NOT NULL,
  api_endpoint  TEXT NOT NULL,        -- e.g. "GET /v1/advice_session/{id}"
  request_body  TEXT,                 -- JSON of the args passed to the client method; NULL if none
  response_body TEXT NOT NULL,        -- JSON.stringify of the parsed response
  timestamp     TEXT NOT NULL         -- ISO-8601 in UTC
);
CREATE INDEX IF NOT EXISTS idx_session_api_calls_session
  ON session_api_calls(session_id);

CREATE TABLE IF NOT EXISTS other_api_calls (
  id            INTEGER PRIMARY KEY AUTOINCREMENT,
  advisor_id    TEXT,                 -- nullable
  investor_id   TEXT,                 -- nullable
  api_endpoint  TEXT NOT NULL,
  request_body  TEXT,                 -- JSON of the args passed to the client method; NULL if none
  response_body TEXT NOT NULL,
  timestamp     TEXT NOT NULL
);
CREATE INDEX IF NOT EXISTS idx_other_api_calls_advisor
  ON other_api_calls(advisor_id);
CREATE INDEX IF NOT EXISTS idx_other_api_calls_investor
  ON other_api_calls(investor_id);
```

Notes:
- `api_endpoint` is a human-readable label like `"GET /v1/advice_session/{id}"` — the templated path, not the substituted URL — so endpoints group naturally when querying.
- `request_body` holds `JSON.stringify` of the arguments passed to the `DeepAlphaClient` method, normalized to a single object (see "Request payload normalization" below). It is `NULL` only when the client method takes no arguments at all.
- `response_body` is always TEXT containing JSON. For the PDF endpoint it stores a small JSON object instead of binary (see PDF section).
- `timestamp` is generated by the app at write time (`new Date().toISOString()`), not via SQLite `CURRENT_TIMESTAMP`, so the recorded time matches the moment the response was processed in Node/Bun.

### Request payload normalization

Client methods have heterogeneous signatures — some take a single options object (`listAdvisors({ email })`), some take positional ids (`getAdviceGoal(sessionId, goalId)`), some take both (`downloadReportPdf(investorId, sessionId, { files })`). To keep `request_body` uniform:

- Single-object args → store the object as-is (e.g. `{"email":"alice@quantfol.io"}`).
- Positional-only args → store as an object keyed by parameter name (e.g. `{"sessionId":"abc","goalId":"g1"}`).
- Mixed positional + options → merge into one object (e.g. `{"investorId":"i1","sessionId":"s1","files":[...]}`).

The exact key names per endpoint are listed in the call-site mapping below.

### Module layout

A new file:

```
db.ts
```

Exports:
- A `Database` handle backed by `data/qap.sqlite`, opened once at module load (see "DB location and lifecycle").
- `recordSessionCall(sessionId: string, endpoint: string, requestBody: unknown | undefined, responseBody: unknown): void`
- `recordOtherCall(ctx: { advisorId?: string; investorId?: string }, endpoint: string, requestBody: unknown | undefined, responseBody: unknown): void`

Both recorders:
- `JSON.stringify` `requestBody` (or write `NULL` if it is `undefined`) and `responseBody`.
- Insert a row with the current timestamp.
- Run synchronously — `bun:sqlite` is sync; no need for a queue.

No changes to `public_api/client.ts`. The recorders are invoked from the call sites in `select_and_lookup.ts` immediately after each `await client.X(...)` resolves.

### Call-site mapping

`req` denotes the normalized request-payload object passed as `request_body`.

| File / function | Client call | Recorder | `api_endpoint` label | `req` shape |
|---|---|---|---|---|
| `handlePick` | `listAdvisors({ email })` | `recordOtherCall({}, ..., req, res)` | `GET /v1/advisor` | `{ email }` |
| `handlePick` | `createInvestor({...})` (no advisor) | `recordOtherCall({ investorId: created.id }, ..., req, res)` | `POST /v1/investor` | `{ name, email, country, investorType }` |
| `handleCreateSession` | `listInvestors({ advisorId, pageSize })` | `recordOtherCall({ advisorId }, ..., req, res)` | `GET /v1/investor` | `{ advisorId, pageSize }` |
| `handleCreateSession` | `createInvestor({...})` | `recordOtherCall({ advisorId, investorId: new.id }, ..., req, res)` | `POST /v1/investor` | `{ name, email, country, investorType, advisorId }` |
| `handleCreateSession` | `createStateSession({...})` | `recordOtherCall({ advisorId, investorId }, ..., req, res)` | `POST /v1/state_session` | `{ advisor_id, investor_id, name, advice_type }` |
| `fetchSessionBundle` | `getAdviceSession(sessionId)` | `recordSessionCall(sessionId, ..., req, res)` | `GET /v2/advice_session/{session_id}` | `{ sessionId }` |
| `fetchSessionBundle` | `getAdviceInformation(sessionId)` | `recordSessionCall(sessionId, ..., req, res)` | `GET /v2/advice_session/{session_id}/advice_information` | `{ sessionId }` |
| `fetchSessionBundle` | `listAdviceGoals(sessionId)` | `recordSessionCall(sessionId, ..., req, res)` | `GET /v2/advice_session/{session_id}/goal` | `{ sessionId }` |
| `fetchSessionBundle` | `getAdviceGoal(sessionId, goalId)` (per goal) | `recordSessionCall(sessionId, ..., req, res)` | `GET /v2/advice_session/{session_id}/goal/{goal_id}` | `{ sessionId, goalId }` |
| `fetchSessionBundle` | `getAdviceTransactions(sessionId)` | `recordSessionCall(sessionId, ..., req, res)` | `GET /v2/advice_session/{session_id}/transactions` | `{ sessionId }` |
| `handleSessionCompleted` | `getReport(investorId, sessionId)` | `recordSessionCall(sessionId, ..., req, res)` | `GET /v1/report/{investor_id}/{session_id}` | `{ investorId, sessionId }` |
| `handleSessionCompleted` | `downloadReportPdf(investorId, sessionId, { files })` | `recordSessionCall(sessionId, ..., req, res)` — see PDF section | `POST /v1/report/{investor_id}/{session_id}/download` | `{ investorId, sessionId, files }` |

Notes:
- `createStateSession` goes into `other_api_calls` because at call time we have `advisor_id` and `investor_id` but not yet a `session_id` — the session is being created by that call. The returned `session_id` will appear on subsequent `session_api_calls` rows.
- Per-goal `getAdviceGoal` produces one row per goal.

### PDF handling

`downloadReportPdf` returns a `Response` whose body is binary PDF bytes. Rather than base64-encoding into TEXT or adding a BLOB column, the PDF continues to be written to disk at `session_<sessionId>.pdf` (existing behavior). The corresponding row in `session_api_calls` records the metadata:

```json
{ "path": "session_<sessionId>.pdf", "size_bytes": 12345 }
```

`size_bytes` is read from `Bun.file(pdfPath).size` after the file is written — this avoids holding the PDF in memory just to measure it and works with the existing `await Bun.write(pdfPath, pdfRes)` pattern.

### Existing JSON dump

The current `session_<sessionId>_data.json` write in `handleSessionCompleted` is removed. SQLite now holds one row per response and is the source of truth. PDF artifacts remain on disk.

### DB location and lifecycle

- File path: `data/qap.sqlite` (relative to repo root). `data/` is created at boot if absent.
- Both `data/` and `data/qap.sqlite*` (including `-wal`, `-shm` if WAL is ever enabled) are added to `.gitignore`.
- The database handle is opened once at module load in `db.ts`. The schema is applied with `CREATE TABLE IF NOT EXISTS` / `CREATE INDEX IF NOT EXISTS` so re-opens are safe.
- No explicit close on shutdown — `bun:sqlite` flushes synchronously per statement.

## Data flow

```
Client → /pick or /session → DeepAlphaClient.X() → response
                                                      ↓
                                              recordOtherCall(...)
                                                      ↓
                                                 other_api_calls

Webhook (qap.advice_session.completed)
   → handleSessionCompleted(sessionId, investorId)
     → fetchSessionBundle, getReport, downloadReportPdf
                                                      ↓
                                            recordSessionCall(...)
                                                      ↓
                                              session_api_calls
                                            (+ PDF file on disk)
```

## Error handling

- API errors (`DeepAlphaApiError` or thrown exceptions) propagate as today. No row is written for a failed call — the recorder only runs after a successful `await`.
- SQLite errors during insert: log via `console.error` and continue. A failed log row must not break the user-facing API flow. The existing handler-level error handling (HTTP 500 / JSON error response) is unchanged.

## Testing

Manual verification is sufficient for this scratchpad:

1. Start the server with a fresh `data/` directory and confirm `data/qap.sqlite` is created with both tables (`sqlite3 data/qap.sqlite ".schema"`).
2. Run through `/pick?user=0`, `/session` POST, and a webhook `qap.advice_session.completed` event end-to-end.
3. Inspect:
   - `SELECT api_endpoint, advisor_id, investor_id, request_body, timestamp FROM other_api_calls ORDER BY id;`
   - `SELECT api_endpoint, session_id, request_body, timestamp FROM session_api_calls ORDER BY id;`
   - For at least one row, confirm `request_body` is non-NULL valid JSON matching the normalized shape from the call-site mapping.
   - One `session_api_calls` row with `response_body` containing `{"path": ..., "size_bytes": ...}` for the PDF endpoint, alongside the actual `session_<id>.pdf` file on disk.
4. Restart the process and confirm prior rows are still queryable.

No automated test suite is added — the project has none today, and adding one is out of scope for this change.

## Out of scope (future work, not part of this spec)

- Logging request headers, response status codes, or timing.
- Logging failed responses.
- A read-side dashboard or query helpers.
- BLOB storage of PDFs inside SQLite.
- Schema migrations for evolving column sets.
